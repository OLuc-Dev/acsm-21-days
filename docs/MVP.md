# MVP do ACSM 21 Days

Este documento fecha o escopo da primeira versão do ACSM 21 Days.

## Objetivo do MVP

Criar uma versão simples, bonita e funcional que prove a experiência principal:

> transformar medo em ação prática por meio de uma jornada guiada de hábitos e check-ins emocionais.

## O que entra no MVP

### 1. Landing page

A landing page deve explicar o produto em poucos blocos:

- promessa principal;
- problema emocional;
- como funciona;
- chamada para iniciar a jornada.

A linguagem deve ser direta e psicológica, sem coaching genérico.

### 2. Onboarding

O onboarding deve coletar:

- nome opcional;
- objetivo principal;
- medo dominante;
- duração da jornada: 7, 14 ou 21 dias.

### 3. Dashboard diário

O dashboard deve mostrar:

- dia atual da jornada;
- desafio do dia;
- ação prática;
- pergunta de reflexão;
- botão de concluir check-in.

### 4. Check-in emocional

O check-in deve registrar:

- nota emocional de 1 a 5;
- se o desafio foi concluído;
- reflexão curta do usuário.

### 5. Progresso

A tela de progresso deve mostrar:

- dias concluídos;
- porcentagem da jornada;
- streak atual;
- média emocional;
- medo trabalhado.

### 6. Resultado final

Ao terminar a jornada, o usuário deve receber um resumo:

- objetivo inicial;
- medo dominante;
- dias concluídos;
- evolução emocional;
- frase final no tom ACSM;
- card compartilhável.

## O que não entra no MVP

Para evitar travar o projeto, a primeira versão não deve incluir:

- login obrigatório;
- pagamento;
- comunidade;
- ranking;
- app mobile nativo;
- IA complexa;
- dashboard administrativo;
- múltiplos idiomas;
- notificações push.

Essas funcionalidades ficam para fases futuras.

## Jornada inicial recomendada

A primeira jornada pode ter 7 dias fixos para prototipação rápida.

Depois, expandir para 14 e 21 dias.

### Exemplo de 7 dias

1. Diagnóstico do medo.
2. Quebra da fuga automática.
3. Ação pequena e desconfortável.
4. Exposição controlada.
5. Disciplina sem motivação.
6. Revisão do padrão antigo.
7. Resumo da reconstrução.

## Critérios de pronto

A primeira versão está pronta quando:

- o usuário consegue iniciar uma jornada;
- os dados persistem ao atualizar a página;
- o usuário consegue fazer check-in;
- o progresso muda visualmente;
- o produto tem identidade ACSM;
- o fluxo funciona no celular;
- o README explica claramente o projeto.

## Prioridade de desenvolvimento

1. Setup do projeto.
2. Design system mínimo.
3. Landing page.
4. Onboarding.
5. Mock de desafios.
6. Persistência local.
7. Dashboard.
8. Check-in.
9. Progresso.
10. Resultado final.
